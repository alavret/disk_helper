# y360_disk — утилита для работы с дисками Яндекс 360

Интерактивный скрипт для получения метаданных файлов и папок на **Общих дисках** и **личных дисках пользователей** организации Яндекс 360.

## Требования

- Python 3.10+
- Доступ к Яндекс 360 API (OAuth-токен с необходимыми правами)
- Для работы с личными дисками: зарегистрированное OAuth-приложение в Яндекс ID с правами доступа к Диску

## Установка

```bash
# 1. Клонировать репозиторий
git clone <repo_url>
cd disk_helper

# 2. Создать и активировать виртуальное окружение
python3 -m venv .venv
source .venv/bin/activate        # macOS / Linux
.venv\Scripts\activate           # Windows

# 3. Установить зависимости
pip install requests python-dotenv

# 4. Создать файл конфигурации
cp .env.example .env             # если есть шаблон, иначе создать вручную

# 5. Заполнить параметры в .env (см. раздел ниже)

# 6. Запустить скрипт
python y360_disk.py
```

## Настройка (.env)

Скрипт читает конфигурацию из файла `.env`, расположенного в той же директории.

### Обязательные параметры

| Параметр | Описание | Пример |
|---|---|---|
| `OAUTH_TOKEN` | OAuth-токен **администратора** организации Яндекс 360 | `y0__xC5sq...` |
| `ORG_ID` | Числовой ID организации в Яндекс 360 | `1234567` |

#### Необходимые права для `OAUTH_TOKEN`

Токен должен быть выписан для аккаунта с доступом к организации и содержать следующие разрешения:

| Разрешение | Назначение |
|---|---|
| `directory:read_users` | Чтение списка пользователей организации |
| `directory:read_organization` | Чтение данных организации |

Для управления сервисными приложениями (пункт 9 меню) токен **дополнительно** должен принадлежать **владельцу** организации (аккаунт `@yandex.ru`) и содержать:

| Разрешение | Назначение |
|---|---|
| `ya360_security:service_applications_read` | Чтение списка сервисных приложений |
| `ya360_security:service_applications_write` | Изменение списка сервисных приложений |

---

### Параметры сервисного приложения

> **Важно.** Параметры `SERVICE_APP_ID` и `SERVICE_APP_SECRET` необходимы **исключительно для работы с личными дисками** пользователей организации (пункт 2 главного меню). Для работы с Общими дисками они не требуются.

Личный диск каждого пользователя Яндекс 360 доступен только от имени этого пользователя. Скрипт получает доступ к ним через механизм **token exchange** (RFC 8693): сервисное приложение обменивает email пользователя на временный OAuth-токен с правами доступа к его диску. Это позволяет автоматически обрабатывать диски произвольного числа пользователей без ручного ввода токенов.

#### Регистрация OAuth-приложения

1. Войдите на [oauth.yandex.ru](https://oauth.yandex.ru) от имени **владельца** организации.
2. Создайте новое приложение. В разделе **Платформы** выберите «Веб-сервисы».
3. Запомните значения **Client ID** (`SERVICE_APP_ID`) и **Client secret** (`SERVICE_APP_SECRET`).
4. Перейдите в **Права доступа** и добавьте следующие разрешения Яндекс Диска:

| Разрешение | Описание |
|---|---|
| `cloud_api:disk.info` | Чтение информации о диске |

5. После заполнения `.env` выполните **настройку сервисного приложения** через пункт `9 → 2` главного меню. Это активирует функционал сервисных приложений в организации и пропишет приложение в список допустимых.

| Параметр | Описание | Пример |
|---|---|---|
| `SERVICE_APP_ID` | Client ID OAuth-приложения в Яндекс ID | `8249314f010b45b8a9c40f597xxxxxx` |
| `SERVICE_APP_SECRET` | Client secret OAuth-приложения | `d6b2411ef68c405aa0d861d3xxxxxx` |

### Параметры файлов

| Параметр | По умолчанию | Описание |
|---|---|---|
| `DISK_RESOURCE_INPUT_FILE` | `input_disk_resources.csv` | Входной файл со списком путей к ресурсам |
| `RESOURCE_OUTPUT_FILE` | `resource_info.csv` | Базовое имя выходного CSV-файла (к имени добавляется временная метка) |
| `USERS_FILE` | `users.csv` | CSV-файл со списком пользователей для поиска по личным дискам |
| `SERVICE_APP_API_DATA_FILE` | `service_app_api_data.json` | Файл для экспорта / импорта данных сервисных приложений |

### Прочие параметры

| Параметр | По умолчанию | Описание |
|---|---|---|
| `VD_HASHES` | — | Список меток (хэшей) Общих дисков через запятую |
| `DRY_RUN` | `false` | Пробный режим (зарезервировано для будущих операций записи) |

**Пример `.env`:**

```dotenv
OAUTH_TOKEN = y0__xC5sqCkqveAAhjj9TUgobS_sRZ2_pJgIDptK3xxxxxxxxxxxxx
ORG_ID = 1234567

SERVICE_APP_ID = 8249314f010b45b8a9c40fxxxxxxxxx
SERVICE_APP_SECRET = d6b2411ef68c405aa0d8xxxxxxxxx

VD_HASHES = ajAWqYhxxxxxxx, 6DE3vqGxxxxxxx, bKIcNTqxxxxxxx

DISK_RESOURCE_INPUT_FILE = input_disk_resources.csv
RESOURCE_OUTPUT_FILE = resource_info.csv
USERS_FILE = users.csv
SERVICE_APP_API_DATA_FILE = service_app_api_data.json

DRY_RUN = false
```

## Форматы файлов

### Входной файл (`DISK_RESOURCE_INPUT_FILE`)

Текстовый файл (кодировка UTF-8 или UTF-8 BOM), по одному пути к ресурсу на строку.

- Строки, начинающиеся с `#`, игнорируются (комментарии).
- Пустые строки пропускаются.
- Допустимы пути в формате Windows (`\`) и Unix (`/`).
- Префикс `<Root>` или `<root>` отбрасывается автоматически.

**Пример:**

```
# Эти строки — комментарии, будут пропущены
#\folder1\file1.txt
#\folder2\file2.docx

# Реальные пути для обработки
/folder3/folder31/file3.docx
\folder4\file4.xlsx
/Folder5/File5.pptx
/FOLDER6/FILE6.docx
```

> Поиск выполняется в два прохода: сначала по точному пути, затем с нечувствительным к регистру сравнением имён компонентов пути.

---

### Выходной файл (`RESOURCE_OUTPUT_FILE`)

CSV-файл с разделителем `;`, кодировка UTF-8. Имя файла автоматически дополняется временной меткой: `resource_info_20260317_181637.csv`.

| Столбец | Описание |
|---|---|
| `name` | Имя файла или папки |
| `path` | Полный путь ресурса в API Яндекс Диска |
| `created` | Дата и время создания (ISO 8601) |
| `modified` | Дата и время последнего изменения (ISO 8601) |
| `md5` | MD5-хэш файла |
| `sha256` | SHA-256-хэш файла |
| `type` | Тип ресурса: `file` или `dir` |
| `size` | Размер в байтах |
| `source` | Для Общего диска — хэш диска (`VD_HASH`); для личного диска — email пользователя |
| `error` | Описание ошибки (если ресурс не найден или возникла неоднозначность пути) |

**Пример:**

```
name;path;created;modified;md5;sha256;type;size;source;error
file4.xlsx;vd:/bKIcNTqxxxxx/disk/folder4/file4.xlsx;2025-11-18T10:35:17+00:00;2025-11-18T10:35:17+00:00;2f673eee...;d91e7b2e...;file;5826;bKIcNTqxxxxxx;
Презентация.pptx;vd:/bKIcNTqxxxxx/disk/folder9/Презентация.pptx;2024-12-01T20:10:26+00:00;...;file;29132;bKIcNTqxxxxx;
```

---

### Файл пользователей (`USERS_FILE`)

CSV-файл (кодировка UTF-8 или UTF-8 BOM) с заголовком. Должен содержать столбец `Email` (регистр не важен).

```
Email
ivanov@company.ru
petrova@company.ru
sidorov@company.ru
```

> Используется при выборе режима загрузки пользователей из файла (`!`) в пункте 2 главного меню.

---

### Файл данных сервисных приложений (`SERVICE_APP_API_DATA_FILE`)

JSON-файл, содержащий список сервисных приложений организации в формате API Яндекс 360. Используется для резервного копирования и восстановления конфигурации.

```json
{
  "applications": [
    {
      "id": "8249314f010b45b8a9c4xxxxxxxxx",
      "scopes": [
        "cloud_api:disk.info"
      ]
    }
  ]
}
```

## Опции меню

### Главное меню

```
1. Получить метаданные ресурсов Общего Диска.
2. Получить метаданные ресурсов из личных Дисков пользователей.
9. Настройка сервисного приложения.
0. Выход
```

| Пункт | Описание |
|---|---|
| **1** | Поиск файлов и папок по входному списку путей на всех Общих дисках, указанных в `VD_HASHES`. Токен авторизации — `OAUTH_TOKEN`. |
| **2** | Поиск файлов и папок по входному списку путей на личных дисках выбранных пользователей. Требует настроенного сервисного приложения. |
| **9** | Переход в подменю управления сервисным приложением. |
| **0** | Завершение работы скрипта (аналог `Ctrl+C`). |

#### Выбор пользователей (пункт 2)

При запуске пункта 2 скрипт запрашивает список пользователей для поиска:

| Ввод | Действие |
|---|---|
| Имя (фамилия), nickname или UID | Поиск конкретных пользователей (через пробел или запятую) |
| `*` | Все пользователи организации |
| `!` | Загрузить список из файла `USERS_FILE` |
| `Enter` (пустой ввод) | Выход в главное меню |

---

### Подменю сервисного приложения (пункт 9)

```
1. Проверить статус сервисного приложения.
2. Настроить сервисное приложение.
3. Удаление сервисного приложения из списка организации.
4. Выгрузить данные сервисных приложений в файл.
5. Загрузить параметры сервисных приложений из файла.
0. Выйти в главное меню.
```

| Пункт | Описание |
|---|---|
| **1** | Проверяет, что приложение присутствует в списке организации, имеет все необходимые права и что токен-обмен работает корректно на тестовом пользователе. |
| **2** | Активирует функционал сервисных приложений в организации (если ещё не активирован), добавляет или обновляет запись приложения с нужными разрешениями. Требует токена **владельца** организации. |
| **3** | Удаляет приложение из списка организации. Если список после удаления пуст — деактивирует функционал. |
| **4** | Экспортирует полный список сервисных приложений в JSON-файл (`SERVICE_APP_API_DATA_FILE` с временной меткой). |
| **5** | Импортирует список сервисных приложений из JSON-файла в организацию. |

## Сценарии использования

### Сценарий 1: получение метаданных ресурсов Общего диска

**Цель:** проверить наличие файлов из списка на одном или нескольких Общих дисках организации и получить их метаданные (хэши, размер, даты).

1. Укажите хэши Общих дисков в `VD_HASHES` файла `.env`.
2. Поместите пути к файлам и папкам в `DISK_RESOURCE_INPUT_FILE` (по одному на строку).
3. Запустите скрипт и выберите пункт **1**.
4. Скрипт последовательно обходит все диски из `VD_HASHES`:
   - **Фаза 1** — прямой запрос по оригинальному пути.
   - **Фаза 2** — нечувствительный к регистру поиск для ресурсов, не найденных в фазе 1.
5. Результаты сохраняются в файл `<RESOURCE_OUTPUT_FILE>_<timestamp>.csv`.

---

### Сценарий 2: получение метаданных ресурсов с личных дисков пользователей

**Цель:** найти файлы из общего списка на личных дисках нескольких сотрудников организации.

#### Предварительная настройка (выполняется один раз)

1. Зарегистрируйте OAuth-приложение в [Яндекс ID](https://oauth.yandex.ru) с правом `cloud_api:disk.info`.
2. Укажите `SERVICE_APP_ID` и `SERVICE_APP_SECRET` в `.env`.
3. Запустите скрипт, перейдите в **9 → 2** («Настроить сервисное приложение»). Скрипт автоматически:
   - активирует функционал сервисных приложений в организации,
   - добавит ваше приложение в список с нужными правами.
4. Убедитесь, что статус **9 → 1** показывает корректную настройку.

#### Выполнение поиска

1. Поместите пути к ресурсам в `DISK_RESOURCE_INPUT_FILE`.
2. Запустите скрипт и выберите пункт **2**.
3. Введите список пользователей (или `*` для всех / `!` для загрузки из файла).
4. Скрипт получит временный токен для каждого пользователя через механизм token exchange и последовательно проверит личный диск каждого из них.
5. В поле `source` выходного файла будет указан email пользователя, на чьём диске найден ресурс.

---

### Сценарий 3: резервное копирование и восстановление конфигурации сервисных приложений

**Цель:** перенести конфигурацию сервисных приложений между организациями или восстановить после сброса.

1. В исходной организации: **9 → 4** — выгрузить список приложений в JSON-файл.
2. Скопировать файл в нужное место, указать путь в `SERVICE_APP_API_DATA_FILE`.
3. В целевой организации: **9 → 5** — импортировать список из файла.

## Логирование

Скрипт ведёт два лога одновременно:

| Поток | Уровень | Файл |
|---|---|---|
| Консоль | `INFO` | — |
| Файл | `DEBUG` | `y360_disk.log` (ротация по 10 МБ, до 5 резервных копий) |

Для диагностики ошибок API обратитесь к файлу `y360_disk.log`.
